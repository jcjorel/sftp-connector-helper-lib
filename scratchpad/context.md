# SFTP Connector Helper Library — Consolidated Specification

Java 21 helper library for AWS Transfer Family SFTP Connector. Public release on GitHub (https://github.com/jcjorel), MIT license, with README.md, docs/, etc.

## Design Principles

- **Fail fast and loud**: throw on any unexpected event. **No fallback mechanisms.**
- **Homogeneous developer experience**: consistent API naming, documentation style, and API paradigms across both features.
- Features are **functionally independent** — no composition/chaining API between them.

---

## Feature 1: Metadata Management around SFTP Connector Async Calls

### Purpose

Generic wrapper for all SFTP Connector asynchronous APIs that stores developer-supplied metadata (JSON) alongside the async call, and retrieves it when the corresponding EventBridge completion event arrives.

### Supported Async APIs

| API | SDK Response ID Field | EventBridge Event ID Field |
|-----|----------------------|---------------------------|
| `StartFileTransfer` | `transferId` | `transfer-id` |
| `StartDirectoryListing` | `listingId` | `listing-id` |
| `StartRemoteMove` | `moveId` | `move-id` |
| `StartRemoteDelete` | `deleteId` | `delete-id` |

The library introspects the SDK response object to extract the correct ID field.

### DynamoDB Table Design

- **Pre-created by the user** (library does not create it).
- **Primary Key**: correlation ID (UUID generated by the library before the SDK call).
- **GSI**: on the job ID field (populated after the SDK call succeeds).
- **TTL**: 1 week by default (configurable).
- **Metadata size limit**: enforced below 256 KB.

### Flow: Storing Metadata

1. Library generates a correlation ID.
2. Library creates a DynamoDB record: `{correlationId (PK), metadata, ttl}`.
3. Library calls the SFTP Connector SDK API.
4. Library introspects the response, extracts the job ID.
5. Library updates the DynamoDB record with the job ID (GSI attribute).
6. Library returns the original SDK response to the caller.

**On SDK call failure (step 3):** The orphan DynamoDB record is left to expire naturally via TTL. The SDK exception is rethrown immediately.

### Flow: Retrieving Metadata (EventBridge event processing)

1. Method receives the EventBridge event as a JSON object (Map/JsonNode).
2. Extracts the job ID from the event (using operation-type → field-name mapping).
3. Queries the DynamoDB GSI by job ID.
4. If record not found (race condition: GSI eventual consistency or job ID not yet written): **throws a specific retryable exception**. The caller is responsible for handling retry (e.g., SQS visibility timeout).
5. If found: returns an enriched object containing the original event + metadata as separate fields.

### Key Behaviors

- `StartFileTransfer` returns a single `TransferId` for up to 10 files. All per-file EventBridge events share the same `transfer-id` → same metadata record. This is by design.
- Both COMPLETED and FAILED EventBridge events are handled identically (metadata is retrieved regardless of status).
- The retryable exception covers two race conditions:
  - Job ID not yet written to the record (SDK call in progress).
  - GSI eventual consistency propagation delay.

## Feature 2: Regex Helper on Directory Listing

### Purpose

Receives an EventBridge directory listing completion event, loads the S3 object containing the remote file/directory listing, applies regex filters, and returns the filtered content.

### Input

An EventBridge event JSON for a completed `DIRECTORY_LISTING` operation. The method extracts:
- `output-directory-path` → S3 bucket
- `output-file-name` → S3 object key

### S3 Object Format (as produced by `StartDirectoryListing`)

```json
{
  "files": [
    {"filePath": "/home/what.txt", "modifiedTimestamp": "2024-01-30T20:34:54Z", "size": 2323}
  ],
  "paths": [
    {"path": "/home/magic"}
  ],
  "truncated": "false"
}
```

### Regex Semantics

| Regex Value | Behavior |
|-------------|----------|
| `null` | Include all entries (no filtering) |
| `""` (empty string) | Exclude all entries |
| `"pattern"` | Keep only entries whose **full path** matches (`Matcher.find()` — pattern found anywhere in the string) |

- One regex for `files` (applied to `filePath` field).
- One regex for `paths` (applied to `path` field).

### Output

Returns the full deserialized structure with filtered content:
- `files` list: filtered by file regex
- `paths` list: filtered by path regex
- `truncated` flag: preserved from original

**No S3 write.** The method only reads and returns filtered content. The user decides what to do with it.

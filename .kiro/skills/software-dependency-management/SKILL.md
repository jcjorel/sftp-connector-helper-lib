The keywords MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD, SHOULD NOT, RECOMMENDED, NOT RECOMMENDED, MAY, and OPTIONAL in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174).

## Critical Constraints

- MUST query authoritative sources only — NEVER guess or use training data for version numbers.
- MUST distinguish stable releases from pre-releases (alpha, beta, RC, M milestones).
- MUST NOT upgrade to pre-release versions unless the user explicitly requests it.
- MUST present findings before applying changes — never auto-upgrade without confirmation.
- MUST verify builds pass after applying upgrades.
- MUST treat Lambda `requirements.txt` files as hash-pinned — regenerate via `uv export` after upgrading.
- MUST regenerate requirements.txt for BOTH Lambdas after any workspace lock update — they share `lambdas/uv.lock`.

## Authoritative Sources

| Ecosystem | Source | Method |
|-----------|--------|--------|
| Maven (dependencies + plugins) | Maven Central | `curl -s "https://search.maven.org/solrsearch/select?q=g:<GROUP>+AND+a:<ARTIFACT>&rows=20&core=gav&wt=json"` — parse JSON, filter out pre-release tokens |
| Maven (when search index lags) | repo1.maven.org | `curl -s "https://repo1.maven.org/maven2/<GROUP_PATH>/<ARTIFACT>/maven-metadata.xml"` — parse `<release>` element |
| AWS SDK for Java v2 | GitHub Releases API | `curl -s "https://api.github.com/repos/aws/aws-sdk-java-v2/releases?per_page=5"` — first non-prerelease entry |
| Python (PyPI) | PyPI JSON API | `curl -s "https://pypi.org/pypi/<PACKAGE>/json"` — read `info.version` |

## Pre-release Detection

MUST filter versions containing any of these tokens (case-insensitive): `alpha`, `beta`, `RC`, `M1`–`M9`, `SNAPSHOT`, `cr`, `preview`.

When Maven Central search returns a pre-release as `latestVersion`, MUST query with `core=gav` and filter the version list to find the true latest stable.

## Dependency Manifest Locations

| Component | File | Format |
|-----------|------|--------|
| Java deps (centralized) | `pom.xml` (root) | `<properties>` block with version variables |
| Java helper library | `helpers/java/pom.xml` | Inherits from parent |
| Integration tests | `tests/integration/java/pom.xml` | Inherits from parent |
| CDK construct | `cdk/pyproject.toml` | `[project.dependencies]` constraints, `[build-system]` pin, `[project.optional-dependencies]` pins |
| Lambda workspace root | `lambdas/pyproject.toml` | uv workspace definition, centralized dev dependencies (`pytest`, `pytest-mock`) |
| Lambda — joiner | `lambdas/joiner/pyproject.toml` | `boto3` version constraint (floor), `hatchling` build-system pin |
| Lambda — event-writer | `lambdas/event-writer/pyproject.toml` | `boto3` version constraint (floor), `hatchling` build-system pin |
| Lambda workspace lock | `lambdas/uv.lock` | Single lock ensuring both Lambdas resolve identical versions |
| Lambda — joiner (output) | `lambdas/joiner/requirements.txt` | Hash-pinned, generated by `uv export` |
| Lambda — event-writer (output) | `lambdas/event-writer/requirements.txt` | Hash-pinned, generated by `uv export` |

## Workflow — Audit Dependencies

1. MUST read all manifest files listed in the Dependency Manifest Locations table.
2. MUST extract each dependency with its current pinned or constrained version.
3. MUST query the appropriate authoritative source for each dependency.
4. MUST filter results to stable versions only (see Pre-release Detection).
5. MUST present a comparison table with columns: Dependency | Current | Latest Stable | Delta (patch/minor/major).
6. SHOULD flag dependencies more than one minor version behind as "notable upgrades."

## Workflow — Upgrade Dependencies

1. MUST run the audit workflow first.
2. MUST ask the user which dependencies to upgrade (all, or a specific subset).
3. MUST apply upgrades according to ecosystem:
   - **Java**: MUST update the version property in root `pom.xml` `<properties>`.
   - **CDK pyproject.toml**: MUST update the version constraint if it is pinned. SHOULD NOT modify `>=` constraints unless the floor needs raising.
   - **Lambda requirements.txt**: MUST update the version constraint in the Lambda's `pyproject.toml`, then regenerate from the workspace lock:
     ```bash
     cd lambdas
     uv lock --upgrade
     cd joiner && uv export --format requirements-txt --no-dev --no-emit-project -o requirements.txt
     cd ../event-writer && uv export --format requirements-txt --no-dev --no-emit-project -o requirements.txt
     ```
     Both Lambdas share a single workspace lock (`lambdas/uv.lock`) — MUST regenerate requirements.txt for both after any lock update.
4. MUST verify the build passes:
   ```bash
   make build
   ```
5. MUST run unit tests:
   ```bash
   make test
   ```
6. If build or tests fail, MUST diagnose and report. MUST NOT silently revert changes.

## Workflow — Upgrade a Single Dependency

1. MUST identify the dependency and its manifest location.
2. MUST query the authoritative source for the latest stable version.
3. MUST apply the version change.
4. MUST run `make build` and `make test`.
5. MUST report the result to the user.

## Version Comparison Logic

MUST parse versions as `MAJOR.MINOR.PATCH` (ignoring qualifiers). MUST classify delta:
- Same major+minor, different patch → `patch`
- Same major, different minor → `minor`
- Different major → `major` ⚠️ (MUST flag for manual review)
